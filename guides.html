Installing Pixel

Pixel requires Python 3.10 or newer and a few dependencies.

  • Python: Install Python 3.10+ from python.org (ensure "Add to PATH" is checked)
  • Install Pixel: Run pip install pixel-automation from the command line
  • Dependencies: Pillow (screen capture), Tesseract OCR (text recognition), and NumPy are installed automatically
  • Tesseract: Install Tesseract OCR separately if not already present (see OCR section)
  • First launch: Run pixel --setup to start the setup wizard and create your first profile

Profiles & Wizard

A profile contains all configuration for a specific automation scenario: detection settings, OCR regions, policy rules, safety parameters, and plugin references.

  • What is a profile? A self-contained config file that defines what Pixel looks for, how it decides, and what actions it takes
  • Setup wizard: Run pixel --setup to create a profile step by step. The wizard walks you through screen region selection, anchor placement, detection config, and safety thresholds
  • Profile fields: Display resolution, ROI coordinates, anchor points, detection method (color markers or template matching), OCR regions, policy rules, safety gate settings, and plugin list
  • Multiple profiles: You can create and switch between profiles for different scenarios. Each profile is a separate JSON file in the profiles/ directory
  • Profile manager: Use the runtime UI to browse, edit, and switch profiles during a session

Calibration & Anchors

Calibration ties Pixel's detection to your specific screen layout. Accurate calibration is essential for reliable operation.

  • Anchors: Stable reference points on screen (e.g., a UI corner or fixed icon) defined by RGB color value, tolerance range, and pixel coordinates
  • Click-capture workflow: Run pixel --calibrate, then click on each anchor point. Pixel records the coordinates and color values automatically
  • Calibration report: After calibration, Pixel generates a report showing anchor confidence, color variance, and suggested tolerance values
  • ROI (Region of Interest): The rectangular area of the screen that Pixel monitors. Defined by top-left and bottom-right coordinates. Smaller ROIs improve performance
  • Resolution matching: Ensure your display resolution and scaling match the profile settings. Calibration is resolution-specific
  • Re-calibration: Re-run calibration whenever you change resolution, scaling, or monitor setup

Detection System

Pixel uses two detection methods that can be used independently or combined for higher accuracy.

  • Color markers: Detect specific RGB color values at defined pixel positions. Each marker has a target color and a tolerance (max delta per channel). Fast and reliable for static UI elements
  • Template matching: Compare a reference image against a screen region using normalized cross-correlation. Returns a confidence score (0.0 to 1.0). Better for complex or variable elements
  • Detection configs: Each detection item specifies the method, target values, confidence threshold, and the ROI sub-region to search within
  • Combined detection: Use both methods together. Pixel merges results and requires all detections to pass before proceeding to the policy engine
  • Detection output: A set of named features (e.g., "health_bar_visible", "menu_open") with boolean or numeric values, passed to the state estimator

OCR

Pixel integrates Tesseract OCR to read text from screen regions and extract numeric or string values for the policy engine.

  • Tesseract install: Download and install Tesseract OCR from the official repository. Add it to your system PATH, or set the path in your profile config
  • OCR regions: Define rectangular screen areas to read. Each region has a name, coordinates, and expected format (text, integer, float, or coordinate pair)
  • Coordinate parsing: Pixel can parse "x, y" formatted text into coordinate values for position-based decisions
  • Preprocessing: Pixel applies grayscale conversion, thresholding, and optional inversion before OCR to improve accuracy on dark UIs
  • OCR output: Named values (e.g., "health_text": "85", "position": [120, 340]) merged into the state alongside detection features

State & Policy Engine

The policy engine takes features from detection and OCR, evaluates rules, and decides what action to take.

  • State estimation: Detection booleans and OCR values are combined into a state dictionary updated each frame
  • Policy rules: Each rule has a condition ("when") and an action ("then"). Conditions reference state keys with comparison operators
  • Action kinds: Click (x, y), key press, key sequence, mouse move, or no-op. Each action can include a delay and a target region
  • Priority: Rules have numeric priority. When multiple rules match, only the highest priority action executes
  • Cooldowns: Each rule can define a cooldown period (minimum time before the same rule fires again) independent of the global cooldown
  • Action output: The chosen action passes through the safety gates before reaching the input layer

Safety Gates

Pixel enforces a 12-layer safety model. Every action must pass all gates before execution. These layers work together to prevent unintended or runaway behavior.

  • Commit threshold: Actions only execute when detection confidence is 99% or higher. Configurable per profile
  • Multi-frame confirmation: The same detection must persist across multiple consecutive frames before triggering an action
  • Cooldown: Global minimum time between any two actions, preventing rapid-fire inputs
  • Hysteresis: Separate thresholds for activation and deactivation to prevent toggling on edge-case detections
  • Recheck frames: After a detection triggers, Pixel re-verifies the state before committing the action
  • Assist mode: When enabled, Pixel highlights the intended action on screen and waits for user confirmation before executing
  • Rate limiting: Maximum number of actions per time window (e.g., 10 actions per minute). Configurable globally and per rule
  • Burst limiting: Maximum consecutive actions without a pause. Forces a cooldown after rapid sequences
  • Auto-disable: Pixel stops automatically if error rate exceeds a threshold or if detections become inconsistent
  • Kill switch: Instantly stops all activity. Default hotkey is F12, configurable in profile settings
  • User-space only: All inputs are sent through standard user-level APIs. No kernel drivers, hooks, or injection
  • Config validation: Pixel validates all profile settings on load and rejects invalid or unsafe configurations

Plugin System

Plugins extend Pixel with reusable detection templates, policy rules, and configuration presets.

  • Plugin structure: Each plugin is a folder containing a manifest file (plugin.json), detection configs, policy rules, and optional template images
  • Manifests: The manifest defines the plugin name, version, author, description, and lists all included configs and templates
  • Plugin configs: Detection settings, OCR regions, and policy rules that merge into your profile when the plugin is activated
  • Template assets: Reference images for template matching, stored in the plugin's templates/ folder
  • Installing plugins: Place the plugin folder in plugins/ and reference it by name in your profile. Pixel loads and validates it on startup
  • Community packs: VIP members can download shared plugin packs from the Downloads page

Runtime UI

Pixel includes a web-based runtime console for live monitoring, diagnostics, and configuration during a session.

  • Web console: Launches at localhost:8080 when Pixel starts. View detection state, policy decisions, and action history in real time
  • Live monitoring: See the current ROI, active detections, OCR values, and safety gate status at a glance
  • Diagnostics: View frame timing, detection confidence trends, OCR accuracy, and error logs
  • ROI adjuster: Fine-tune your region of interest visually through the console without editing config files
  • Profile manager: Switch between profiles, edit settings, and reload configuration without restarting Pixel

Telemetry & Replay

Pixel logs all events for review, debugging, and audit purposes.

  • Event logging: Every detection, policy decision, safety gate result, and action is recorded in JSONL format with timestamps
  • Replay analysis: Replay a session step-by-step to see exactly what Pixel detected, decided, and did at each frame
  • Audit input: Review all inputs sent by Pixel, including coordinates, key presses, timing, and the detection state that triggered each action
  • Replay summary: Generate a summary report with action counts, detection accuracy, safety gate hit rates, and session duration
  • Log location: Telemetry logs are stored in the logs/ directory, organized by session date and profile name

Troubleshooting

Common issues and their solutions:

  • Pixel won't start: Verify Python 3.10+ is installed and on PATH. Run pip install --upgrade pixel-automation
  • OCR not working: Ensure Tesseract is installed and the path is set correctly in your profile or system PATH
  • Detection not matching: Re-calibrate ROI and anchors. Check that your resolution and scaling match the profile
  • Poor accuracy: Increase the commit threshold, add multi-frame confirmation, or switch to template matching for variable elements
  • Plugin validation errors: Check the plugin manifest for missing fields. Run pixel --validate-plugin <name> for details
  • Profile won't load: Run pixel --validate-profile <name> to check for config errors. Common issues: missing anchor data, invalid ROI coordinates, or referencing a missing plugin
  • Runtime UI not loading: Check that port 8080 is not in use. Try pixel --ui-port 8081 to use an alternate port
  • Need more help? Visit Support or the Client Support forum
